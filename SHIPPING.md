How this PWA becomes a paid App Store app. Generated by a research+verify workflow
(`ios-shipping-playbook`, 2026-06-16), then spot-corrected against the real machine state.

**Confidence note:** §1 (package manager) was adversarially verified. §§2–6 are
researched but their independent verify pass died on transient API errors, so treat
them as *solid but single-sourced* — confirm version pins / exact values with a quick

**Verified machine state (Jun 16):** Capacitor `8.4.0`, SPM layout (`ios/App/CapApp-SPM/Package.swift`,
no Podfile, no `.xcworkspace`), Xcode **26.5**, system Ruby 2.6.10, `pod` not installed.

Capacitor 8 requires **Xcode 26.0+** and an **iOS 15.0** deployment target — these gate everything.

## 1. Package manager — SPM, already done ✅

Stay on **Swift Package Manager**. The project is already SPM-based (`npx cap add ios --packagemanager SPM`).
**CocoaPods is a non-issue** — do not install it, do not fight system Ruby 2.6. (The old
"blocked on CocoaPods" note is dead.)

- SPM is Capacitor 8's **production default** (announced Dec 2025) — not the experimental
  Cap-6 version.
- You **cannot mix** SPM and CocoaPods. A future CocoaPods-only plugin would force a full
  migration — but RevenueCat (IAP) ships first-class SPM, so that's unlikely.
- **`ios/debug.xcconfig` = `CAPACITOR_DEBUG = true` (23 bytes, verified correct).** The
  playbook's "empty 0-byte file" warning was a false alarm — ignore it.
- After adding any plugin: `npx cap sync ios`, then Xcode → **File → Packages → Reset
  Package Caches** if dependencies don't resolve.

**Fallback (you won't need it):** if a future plugin is CocoaPods-only, install a modern
Ruby via rbenv (no Homebrew needed) — `git clone rbenv + ruby-build`, `rbenv install 3.3.6`,
`gem install cocoapods`. Never use system Ruby 2.6.

## 2. Signing & device — free team to smoke-test, $99 program for everything real

**Free Personal Team:** quick on-device test only. 7-day cert, ~3 apps, and **cannot enable
Push, App Groups, Associated Domains, or Sign in with Apple.**

**Paid Apple Developer Program ($99/yr):** required for TestFlight, native push, Universal
Links, and IAP sandbox testing. Because LOCKED IN needs push + IAP, **plan to enroll early** —
most real testing can't happen on the free team.

### Path A — Free (cable install)
1. Xcode → Settings → Accounts → ＋ → Apple ID.
2. App target → Signing & Capabilities → Automatically manage signing → Team = Personal Team.
   If "Failed to register bundle identifier," change `com.lockedin.app` → `com.ianpalsgaard.lockedin`.
3. iPhone (iOS 16+): Settings → Privacy & Security → **Developer Mode** → ON → restart.
4. Cable, unlock, Trust This Computer → pick device in Xcode → Run.
5. First launch blocked → phone Settings → General → VPN & Device Management → your Apple ID → Trust.
6. Cert expires in 7 days; re-Run to refresh.

### Path B — Paid → TestFlight
1. Enroll: https://developer.apple.com/programs/enroll/ (~24–48h). **Decide entity vs.
   individual enrollment first** — see §6.9 (health-adjacent apps).
2. App Store Connect → New App (register bundle ID first).
3. Xcode destination = Any iOS Device → bump Version+Build → **Product → Archive** →
   Distribute → Upload.
4. TestFlight: Internal = 100 testers, no review; External = 10k, first build needs Beta App
   Review. Builds expire 90 days after upload.

## 3. Auth deep-link — the fix that makes login work on device

The magic link opens Safari, not the app, because it points at an https Site URL the
WKWebView never sees. **Do not** rely on the PKCE access-token-in-URL flow (the verifier
lives in WebView storage, not Safari's). Use stateless **`token_hash` + `verifyOtp`** over a
**custom URL scheme**. This works on the *free* team (custom schemes don't need Associated
Domains; Universal Links do — that's a later polish).

1. `npm i @capacitor/app && npx cap sync ios` — pin `@capacitor/app` to 8.x (`npm ls @capacitor/core @capacitor/app`).
2. `src/lib/supabaseClient.js` auth options: `{ persistSession: true, autoRefreshToken: true, detectSessionInUrl: false, flowType: 'pkce' }`.
3. `src/components/Login.jsx`: `emailRedirectTo: isNative ? 'com.lockedin.app://login-callback' : window.location.origin` (`isNative = Capacitor.isNativePlatform()`).
4. **Free tier — template is LOCKED** (editing needs custom SMTP). So *don't* touch the
   email template. Leave the default `{{ .ConfirmationURL }}` link. With `flowType: 'pkce'`,
   Supabase's https verify endpoint redirects back to the custom scheme with `?code=...`,
   which `nativeAuth.js` exchanges via `exchangeCodeForSession`. (The token_hash route from
   the old plan needs a custom template — revisit it only if you add SMTP later.)
   - **Requires** `Site URL` set to your https Vercel domain (so `ConfirmationURL` is a safe
     https link, not Go-sanitized to `#ZgotmplZ`).
5. Dashboard → Auth → URL Configuration → Redirect URLs: add `com.lockedin.app://login-callback`,
   `com.lockedin.app://**`, `http://localhost:5173/**`, `https://<vercel-domain>/**`.
   **Known free-tier risk:** Gmail/email link-scanners can pre-consume the one-time link →
   intermittent "token expired." Robust fix is the **Email → Website → App** bridge (a tiny
   page on the Vercel domain with a "Open app" button that `window.location = com.lockedin.app://...`).
   Skip it for the first smoke test; add it if logins fail intermittently.
6. `ios/App/App/Info.plist` → add `CFBundleURLTypes` with scheme `com.lockedin.app`
   (Xcode: App target → Info → URL Types). The Capacitor deep-link guide omits this for
   custom schemes — add it manually.
7. `appUrlOpen` listener early in `src/App.jsx` (or `main.jsx`):
   ```js
   import { App } from '@capacitor/app';
   App.addListener('appUrlOpen', async ({ url }) => {
     const u = new URL(url);
     const token_hash = u.searchParams.get('token_hash');
     const type = u.searchParams.get('type');
     const code = u.searchParams.get('code');
     if (token_hash) await supabase.auth.verifyOtp({ token_hash, type: type || 'email' });
     else if (code)  await supabase.auth.exchangeCodeForSession(code); // future OAuth
     // also: u.searchParams.get('face') → preserve Morning/Examen/Offerings routing
   });
   ```
   The existing `onAuthStateChange` in `AuthGate.jsx` then fires `SIGNED_IN`.
8. Test on a real device; open the email **on the same device**.

**Watch-out:** confirm the template emits `type=email` vs `type=magiclink` — a mismatch gives
"Token has expired or is invalid" even with a valid hash. Confidence: HIGH on architecture,
MEDIUM on the exact `type` value (one device test confirms).

## 4. Service worker — disable it for the native build

Your `src/sw.js` **cannot register** in the native iOS WKWebView (Capacitor serves from
`capacitor://localhost`; `register()` requires http/https). It's dead code on iOS, not a
conflict. Disable VitePWA for the native build; keep it fully intact for web/Vercel/PWA + Android.

1. `package.json`: `"build:ios": "CAP_BUILD=1 vite build"`.
2. `vite.config.js`: `const isNative = process.env.CAP_BUILD === '1';` → VitePWA `disable: isNative`.
3. Runtime guard before any SW/push code: `if (Capacitor.isNativePlatform()) { /* skip register + subscribe */ }`.
4. Native build: `CAP_BUILD=1 npm run build && npx cap sync ios`.
5. **Do not** enable `WKAppBoundDomains` (breaks plugin injection + the magic-link external flow).

## 5. Push — move native to APNs, keep Web Push for web only ($99-gated)

Web Push does **not** work in the Capacitor iOS WKWebView (Apple confines iOS Web Push to
home-screen-installed PWAs). Native must use **APNs via `@capacitor/push-notifications`**.
Both coexist, fanned out from the Supabase edge functions. Requires the paid team.

1. Add a `platform` column (`'webpush' | 'apns'`) to `push_subscriptions`.
2. `npm i @capacitor/push-notifications` (optionally `@capacitor/local-notifications` for
   on-device reminders with no server — could cover Guardian nudges short-term).
3. Xcode: App target → +Capability → Push Notifications; +Background Modes → Remote notifications.
4. `AppDelegate.swift`: forward the APNs token via `.capacitorDidRegisterForRemoteNotifications`
   (raw APNs token, no Firebase needed).
5. JS gated on `Capacitor.getPlatform() === 'ios'`: `requestPermissions` → `register` →
   on `registration`, POST `{ token, platform: 'apns' }`.
6. Server APNs sender beside `supabase/functions/_shared/push.ts`: ES256 JWT signed with the
   APNs `.p8` (refresh < 1h), POST `https://api.push.apple.com/3/device/<token>` over HTTP/2,
   `apns-topic: com.lockedin.app`. New secrets: `APNS_KEY/KEY_ID/TEAM_ID/BUNDLE_ID`.
7. Refactor `sendPushToUser` into a dispatcher (webpush vs apns); callers `referee` and
   `counsel` stay unchanged. Prune on 410.

**Open item:** validate Deno edge HTTP/2 + ES256 before relying on it.

## 6. App Store review — four landmines; ship as a discipline coach, not a medical product

1. **Subscriptions = Apple IAP (3.1.1, non-negotiable).** RevenueCat wraps StoreKit (SPM).
   No Stripe/web paywall in-app, no in-app link to web payment. Mention web pricing only via
   email, outside the app.
2. **Commission:** enroll in the **Small Business Program** (<$1M/yr) → **15%** from year one.
3. **Subscription design (3.1.2):** ongoing value (lean on the AI planner), show full price +
   period, working **Restore Purchases**, Terms + Privacy links, purchase completable in sandbox.
4. **Recovery module (1.4.1):** never show/describe/link explicit content; no medical claims
   ("treats/cures addiction"); frame as wellness/self-help; visible disclaimer ("not a
   substitute for professional treatment"); keep survey-gated.
5. **Faith module (1.1.5):** allowed; keep accurate, opt-in, non-inflammatory; don't make the
   core paid app require it.
6. **AI guidance (1.2 / 5.6.3):** moderate outputs (block sexual/self-harm/medical-diagnosis);
   route self-harm language to real resources (988); factor into age rating (may push to 17+).
7. **AI data sharing (5.1.2, updated Nov 2025 — NEW + ENFORCED):** if journal/survey/recovery
   text goes to a third-party AI (Anthropic/Gemini), you must (1) **name the provider**,
   (2) get **explicit consent BEFORE** sending, (3) allow revoke. This is critical for our
   recovery + faith survey data.
8. **Partner texting (5.1.2 — highest concrete rejection risk):** must be **double-opt-in** —
   the partner explicitly accepts before you store their info or text them. Never auto-text a
   typed number; prefer push/email to an installed partner, or the system share sheet.
9. **Privacy (5.1.1):** in-app + listing privacy policy; **in-app account deletion** (build it
   on Supabase auth); accurate privacy nutrition label; no Contacts access.
   **5.1.1(ix): sensitive/health-adjacent apps should be submitted by a legal entity, not an
   individual — decide before enrolling (§2).**
10. **Metadata (4.3, tightened June 2026):** position as a substantive AI discipline planner,
    not a thin habit-timer; no explicit words/imagery in screenshots.

What keeps this off a 95 is the v1.1 attack surface the plan depends on — the shame-screen only covers InsightCard (the app's core promise to recovery users has a hole), the Opus endpoints have no rate/cost ceiling, and the waitlist/referee/Strava paths carry unauthenticated-write, broken-access-control, and unsigned-OAuth-state gaps that must close before the backend resumes.

**Strengths:**

The core sync schema is genuinely well-designed: thin-key + JSONB + a server-side LWW guard matches the local-first client exactly, and the claimed-vs-witnessed RLS split (owner-write sync tables vs select-only checkpoints/ui_layouts/stakes_history vs no-policy strava_accounts/ai_runs) is disciplined and correctly implemented. But two landmines keep this out of the 85+ band: the waitlist_rank view in 0007 bypasses RLS and would expose every signup email to anyone holding the anon key the day the waitlist migration runs, and 0008's CHECK constraints (terminal/zen/night_ops, financial/social/friction/none) no longer match what Onboarding.jsx actually writes (split_book/lamplight/carbon, witness) — so the moment v1.1 resumes, every profile upsert fails wholesale and the survey silently never persists. - The LWW design is coherent end-to-end: client stamps updated_at, the lww_guard trigger (0001) refuses stale UPDATEs server-side, sync.js mergeInto applies the same rule on pull, and upsert onConflict keys (user_id / id / user_id,k) exactly match the PKs and unique indexes — I verified every slice in SLICES against its table.
- Privacy by construction where it matters most: the handover slice (raw drafts from a recovery user base) is structurally excluded from SLICES so it can never sync (storage.js:69–80), and 0005 push_subscriptions has per-device (user_id, endpoint) uniqueness plus a server-authoritative touch trigger instead of trusting client clocks.
- Migrations are ordered, idempotent, and honestly annotated: 0010 guards publication adds against pg_publication_tables, 0011 unschedules before rescheduling and reads secrets from Vault instead of hardcoding, and deploy-gated migrations are explicitly labeled as safe no-ops.

## local-first-migration — 79/100

The sync engine (src/lib/sync.js + 0001/0004) is a genuinely well-architected LWW core: sidecar stamp/dirty bookkeeping keeps app data pristine, pull-then-push cold-start correctly adopts months of unstamped offline history as local-only puts, and v1 shipping with zero server identity means there is no orphan-account problem to unwind. But the hard 10% of the migration story is missing or wrong: the collection tables use a globally-unique text PK instead of (user_id, id), timestamp comparison mixes toISOString 'Z' with PostgREST '+00:00' formats lexically, whole-blob singleton LWW means the second device to claim an account silently loses its entire streak history, the native auth path was deleted with no replacement, and no code path exists to bridge lockedin:__survey into user_profile.survey — the gate the entire v1.1 Architect plan depends on. Nothing is deployed yet, so every foreclosure is still cheaply fixable — which is exactly why the schema fixes should land now.

**Strengths:**
- Cold-start adoption is already correct in outline: sync.js:276-291 treats unstamped local records as local-only, keeps them, and queues pushes — a device-only book hydrates an empty account without duplication, and idempotent upserts (sync.js:12-14, onConflict) make re-pushes safe.
- Sync bookkeeping lives in its own lockedin:__sync sidecar (sync.js:46-71), so app data shapes and validate.js never know sync exists — the v1→v1.1 data format is already sync-ready with zero migration of the slices themselves.
- Privacy-by-construction exclusions are deliberate and structural: handover (raw relapse confessions) is absent from SLICES (storage.js:71-81), witnessed checkpoints flow through a read-only path (record.js) instead of the read-write engine, and exportAll strips drafts (storage.js:188).
- v1 has no anonymous auth and no server identity at all — the cleanest possible bootstrap position: v1.1 can adopt signInAnonymously→linkIdentity or straight Sign in with Apple with no legacy account graph to merge.
- Server-side lww_guard trigger (0001:20-35) plus owner-only RLS on every synced table means a buggy or malicious client cannot resurrect stale rows or touch another user's data.

## sync-protocol — 75/100

The happy-path protocol is genuinely well designed — idempotent keyed upserts, a clean sidecar dirty queue, fail-soft everywhere, server-side LWW guard plus owner-only RLS applied uniformly across all ten tables. But as a multi-device protocol it has two lose-the-book-class defects (unpaginated pulls feeding an absence-means-deleted rule; a sync sidecar not scoped to the auth uid), no tombstones so deletes cannot be ordered against edits, unhandled clock skew that lww_guard converts into silent divergence, and a realtime design (0010 + schema-wide subscription) that fans every user's delete into a full 10-table pull on every connected client. None of this can hurt anyone while Supabase is paused and v1 has no accounts, which is exactly why all of it must be fixed before v1.1 flips the switch.

**Strengths:**
- Correct idempotent mapping: SLICES kinds match 0001 primary keys exactly (settings/streak/reading on user_id, collections on id, composites on (user_id,k)); onConflict() targets are right, so re-push can never duplicate (sync.js:118-133 vs 0001:39-96).
- Fail-soft discipline is real: every entry point no-ops when unconfigured/sessionless/offline (ready(), sync.js:146-152), per-table pull errors are skipped not fatal (sync.js:240-243), and a failed push leaves the dirty queue intact for retry (sync.js:221-223) — no path locks the UI out.
- mergeInto's pending-edit protection is correct LWW: a dirty local record at-least-as-new as the server's survives the pull (sync.js:266), and local-only records created offline are re-queued for push rather than dropped (sync.js:286-290).
- Server-side enforcement, not client trust: lww_guard trigger + owner-only RLS created in one loop over every synced table (0001:104-125), wellness follows the identical pattern (0004), and 0010's publication list mirrors SLICES exactly with idempotent pg_publication_tables guards.
- Pull-then-push cycle ordering (sync.js:298-303) is right for fresh-device hydration, and _internals exposes the pure translation/merge layer so the protocol is unit-testable without a live backend (245+ tests green).

- **Scope the sync sidecar to the auth uid; clear it on account change** — META_KEY 'lockedin:__sync' (src/lib/sync.js:46) is global, and onAuthStateChange (sync.js:363-372) never resets meta on sign-out or uid change. Scenario: device synced under account A (or restored from an iCloud backup of a synced device), user signs into account B → every local record carries a stamp, so previouslySynced is true, B's first pull returns zero rows, and mergeInto's absence-means-deleted rule (sync.js:278-284) wipes the entire local book. - **Add tombstones; deletes currently bypass LWW entirely** — pushNow executes unconditional .delete().eq(...) (src/lib/sync.js:209-217) — the lww_guard trigger only fires on insert/update (0001:119-123), so a device offline for months that queued a 'del' destroys a row another device has since edited with a months-newer updated_at (stale delete beats fresh edit). Conversely, an offline 'put' resurrects a genuinely newer delete on pull because absence carries no timestamp to compare (sync.js:266 has nothing to check against). Both are unfixable without ordering deletes: add a deleted_at tombstone column (or tombstone table) written instead of DELETE, include it in pulls, and garbage-collect after a retention window. - **Clamp client clock skew server-side in lww_guard** — updated_at is pure client wall-clock (nowISO(), src/lib/sync.js:42,159). A device with a clock set months ahead stamps rows that permanently win LWW on every other device; after the clock corrects, that device's own legitimate writes are rejected by lww_guard — which returns OLD (0001:31), so PostgREST reports success, the client clears the dirty flag (sync.js:207) believing it pushed, and local/server silently diverge until the skewed future stamp is eventually exceeded (possibly months). Comparing '…56.789Z' against '…56.78912+00:00' compares 'Z' to a digit, so sub-second races and equal instants resolve by character codes, not time — wrong winners exactly in the realtime echo window. Fix: convert both sides with Date.parse at the comparison boundary (or normalize serverAt to toISOString() before storing). _[src/lib/sync.js]_
- **Don't clear a dirty flag re-set during the push round-trip** — pushNow snapshots the queue, awaits the network, then deletes dt keys unconditionally (src/lib/sync.js:205-207 for puts, 214-217 for dels). If the user edits record k during the await, onLocalWrite re-marks dt[k]='put' with a newer stamp — and the post-await delete erases that flag, so the newer edit is never pushed until some unrelated future write to the same slice dirties the queue again (local stamp > server stamp means pulls keep the divergence alive indefinitely). Fix: capture st[k] into the snapshot when building rows and only delete dt[k] if st[k] still equals the snapshotted stamp. _[src/lib/sync.js]_
- **Realtime: per-user filters, delta pulls, and a subscribe status handler** — Three problems with the 0010 + sync.js:323-331 design. (1) The channel listens schema-wide with no filter; Supabase postgres_changes cannot RLS-filter DELETE events (the row is gone — only the old PK is delivered), so every connected client receives every user's deletes and responds with a full 10-table pullAll: cross-user PK metadata leak plus a thundering-herd pull storm at any real user count. Add filter: `user_id=eq.${uid}` per table (collections need user_id in the event — they have the column). (2) The client's own push echoes back through realtime and triggers a full re-pull of all ten tables after every write burst, because pullAll has no .gt('updated_at', meta.pulledAt) delta — wire pulledAt in (safe for upserts even before tombstones exist). (3) .subscribe() has no status callback (sync.js:330), so CHANNEL_ERROR/TIMED_OUT silently kills realtime for the rest of the session — handle status and re-subscribe with backoff. _[src/lib/sync.js, supabase/migrations/0010_realtime.sql]_
- **Give collections a composite primary key (user_id, id)** — income/runs/tasks use a globally-unique client-generated text id as the sole PK (0001:56-73). Primary path is crypto.randomUUID (src/lib/store.jsx:20) which is fine, but the documented fallback is Date.now().toString(36)+Math.random().slice(2,11) (store.jsx:21) — cross-user collisions become possible, and an upsert colliding with another user's row fails permanently under RLS (ON CONFLICT DO UPDATE against an invisible row errors), wedging that slice's dirty queue forever with only a console.warn. Before v1.1: migrate the PK to (user_id, id) and change onConflict('collection') from 'id' to 'user_id,id' (sync.js:126) — also makes the pushNow collection delete (sync.js:211) self-scoping instead of RLS-dependent. _[supabase/migrations/0001_phase3_sync.sql, src/lib/sync.js]_

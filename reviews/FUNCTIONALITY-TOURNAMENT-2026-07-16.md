## monetization-correctness — 87/100

The register core is genuinely strong: restore() verifies ownership before granting (checked/!checked split is load-bearing and tested), refreshEntitlement on mount/resume makes lapse and winback real, revocation/expiration/refund guards match the installed plugin's types, and wipeAll correctly spares the entitlement. I found no code path in a correctly-built store binary that leaks an unearned entitlement. What keeps it out of the 90s: the Paywall tells a paying subscriber "No prior purchase found" on a transient verify failure, the analytics funnel mislabels winback re-purchases as trial_start and never implements its own promised subscribe counter, one paywall bullet sells an adaptive-protocol behavior that free users already get, and nothing stops a forgotten VITE_TESTER=1 in .env.local from shipping a store build where every plan tap grants a free mock trial.

**Strengths:**
- restore() never grants without verified ownership and never revokes on a transient error — the { checked:false } vs { checked:true, productId:null } split (purchases.js:205-228, 265-272) is exactly right and pinned by 7 native tests (purchases.native.test.js:33-89)
- refreshEntitlement wired on App mount + visibilitychange (App.jsx:148-155) with a TESTER guard so the mock clock isn't expired every boot — cancelled subs actually lock, winback is actually reachable, and sweep.spec.js:69-80 proves the lapsed paywall renders
- applyOwnership treats "previously owned, Apple now owns nothing" as expired (winback) vs never-owned none (purchases.js:235-248), and readOwnership rejects revocationDate / subscriptionState / past expirationDate so a refunded sub cannot keep the coach (purchases.js:211-222)
- Gate coverage is complete and consistent: drift sentinel + protocol intelligence (GuardianPanel.jsx:71,139), Sunday review (WeeklyReview.jsx:17,76), and Counsel margin notes (widgets.jsx:483-486) all degrade to CoachGate; the book surfaces (urge protocol, record, briefing) deliberately never touch purchases.js
- Dev/test seams are properly fenced: devClearEntitlement/devMarkLapsed/devReloadCache are DEV-gated no-ops in production (purchases.js:352-368), the DEV mock branch sits after the native path so unit tests reach StoreKit code, and wipeAll spares lockedin:__coach (storage.js:210-215)

## offline-and-sync — 86/100

The fail-soft skeleton is genuinely sound: unconfigured collapses to clean synchronous no-ops, and configured-but-paused boots with zero blocking network in v1 (getSession/getUser resolve locally with no session, layout renders instantly from cache/default, push failures leave the persisted dirty queue intact). But the sync engine has two real correctness defects lying in wait for v2 — an in-flight-edit race that silently drops writes from the dirty queue, and delete-mirroring that will reap local records if a months-paused project comes back with an empty or backup-restored database — and the two render-blocking awaits (RequireSurvey's probe, Onboarding's finish) have no timeout, so any persisted session on a flaky connection hangs the boot screen indefinitely. Nothing here harms a v1 user today, which is exactly why it scores solid-with-gaps rather than shippable-tonight: the dormant engine's failure modes are unpinned by tests and would fire the day sync turns on.

**Strengths:**
- Unconfigured fail-soft is airtight: one flag (supabaseClient.js:40), a null client, and every entry point in sync.js/layout.js/counselClient.js/architectClient.js checks it before touching the network — the app is provably whole with no backend.
- Configured-but-paused boot never blocks: sync.start's getSession (sync.js:355) and RequireSurvey's getUser with no persisted session resolve locally without a fetch, useLayout paints from cache/default before its one doomed GET settles (layout.js:56-91), and no paused-endpoint failure can throw into the UI (postgrest/functions errors are returned, not thrown).
- Resume-after-months skeleton is right: pull-before-push (sync.js:298-303), idempotent keyed upserts with per-record client stamps, LWW enforced on both merge (mergeInto:266-272) and a server guard trigger, and mergeInto:286-290 re-queues local-only records so offline months of writes push up on first sign-in.
- Push failure handling is honest: the dirty queue is a persisted localStorage sidecar (META_KEY, sync.js:46-63) left fully intact on any error (sync.js:221-223), retried on next write/online/login, with the pushing/pushQueued latch preventing overlapping pushes.
- The UI never lies in v1: status chip reads ON DEVICE when configured-but-signed-out (App.jsx:45), the queued-n chip only appears signed-in+offline (App.jsx:61-68), and CommandPanel says “every entry stays on this device” (CommandPanel.jsx:69-76).

**BEFORE LAUNCH:**
- **RequireSurvey's DB probe has no timeout — boot can hang on “Syncing profile…”** — src/components/RequireSurvey.jsx:42-69: when no local survey exists and Supabase is configured, the gate awaits supabase.auth.getUser() then a user_profile select with no AbortSignal or timeout. With no persisted session getUser resolves locally (fast), but any persisted session (Ian's dev/TestFlight devices today; any auth remnant or v2 account) on a flaky connection blocks the ENTIRE app — including the HELP/crisis nav, which only mounts past this gate — for the OS-level fetch timeout (60s+). Fix: use getSession() (local, no round-trip) instead of getUser(), and wrap the profile select in Promise.race with a ~4s timer that resolves to setState('need'). The error→'need' fallback already exists at line 63-68, so the timeout path is one line of the same tradeoff the file already documents. _[boot, RequireSurvey gate, crisis path]_
- **Onboarding finish() awaits un-timeboxed network calls on the Processing screen** — src/pages/Onboarding.jsx:750: finish() does `await writeProfile(...)` whose getUser+upsert (lines 627-632) have no timeout; buildInitialLayout at 757 is try/caught but still awaited. In v1 (no session) both short-circuit fast, but with a session on a flaky/paused backend the user completes the whole intake and then hangs forever on the Processing screen — the survey IS already saved locally by applyLocally (line 748), so the hang is pure lost trust. Fix: Promise.race writeProfile/buildInitialLayout against a ~5s timer (the 1300ms `floor` at line 749 already shows the pattern) so `window.location.href = '/'` always runs. _[Onboarding]_
- **invokeCounsel is not session-gated — the shipping build fires doomed POSTs at the paused project** — src/lib/counselClient.js:14 gates only on isSupabaseConfigured, unlike architectClient.js:16-22 which also requires a session. In the shipping mode (keys baked, project paused, no accounts) every LiveDeck mount with a drift pattern — and every pattern/partner dep change (LiveDeck.jsx:261-272) — fires a functions.invoke POST that can only fail: radio wakeups, console noise, and latency-long pending promises for zero possible payoff. Fix: copy architectClient's getSession() gate into invokeCounsel and early-return null when navigator.onLine === false; behavior for real signed-in users is unchanged. _[LiveDeck, Counsel]_

**FUTURE:**
- **pushNow race: an edit made during an in-flight upsert is dropped from the dirty queue** — src/lib/sync.js:202-207: puts are built from st[k] BEFORE `await supabase.upsert` (line 205); if onLocalWrite (line 155) fires during that await it re-stamps st[k] and re-marks dt[k]='put', but line 207's `delete dt[serverKey(s,row)]` then wipes the NEW dirty flag. The pushQueued re-push (226-229) finds an empty queue; on later pulls the local stamp is newer than the server's so mergeInto:266 keeps local without re-queueing — the edit silently never syncs until that exact record is touched again. Same race on the delete loop (209-217) if a record is re-created mid-delete. Fix: only clear the flag if the stamp is unchanged since build — `const key = serverKey(s,row); if (st[key] === row.updated_at) delete dt[key]` (row.updated_at IS the build-time stamp via rowFor, line 118-123), and capture st[k] before each delete similarly. Dormant in v1 (no session can exist) but must land before sync ever enables. _[sync engine]_
- **mergeInto mirrors deletes on an empty-but-successful pull — resume-after-months can reap local data** — src/lib/sync.js:276-285: any previously-synced (st[key] set, dt clear) non-singleton record absent from a pull is deleted locally, with absence as the only delete signal. A project paused for months and restored from backup (Supabase's documented pause/restore path), a migration reset, or a sign-in under a different account returns empty/partial rows with NO error — and the client deletes months of local tasks/sprints/checklist days to match. This is the single worst answer to “what happens when sync resumes after months.” Fix: never mirror deletes on a pull where a table returned zero rows while local holds previously-synced records (treat as “server amnesia,” re-queue local as puts via the existing 286-290 branch), or move to explicit tombstones server-side. Also scope st/dirty meta per user_id so an account switch can't cross-contaminate. _[sync engine, data integrity]_
- **pullAll stamps pulledAt even when every table's select failed** — src/lib/sync.js:239-247: each per-table error only warns and continues, then line 246 unconditionally sets meta.pulledAt = nowISO(). Against a paused project every select fails yet status().pulledAt advances, and CommandPanel.jsx:72-74 renders it as a real “Sync: HH:MM” to a signed-in user whose data never moved. Fix: track a success flag in the loop and only stamp pulledAt if at least one select succeeded. _[CommandPanel, sync status]_
- **useLayout re-parses the layout cache on every render** — src/lib/layout.js:67: `const initial = pickInitial()` runs in the hook body, so every LiveDeck render (which re-renders on each store change) does a localStorage.getItem + JSON.parse + full normalizeLayout of the deck payload, then throws the result away (useState only reads it once). Fix: `const [initial] = useState(() => pickInitial())` (or useMemo with []) and derive both initial states from it. _[LiveDeck, performance]_
- **Sync sidecar meta grows unboundedly in the shipping build and stamps are never pruned** — src/lib/sync.js:155-171: with keys baked but no session ever possible (v1), sync.start still subscribes storage, so every write diffs, stamps, dirty-marks, JSON-serializes the whole meta (saveMeta) and arms an 800ms timer for a pushNow that no-ops at ready(). Daily-keyed composites (sprints/checklist/wellness) add stamps+dirty entries forever, and st entries for server-deleted records are never garbage-collected (the 276-285 branch only visits keys still present locally). Fix: skip dirty-marking (keep stamps if you want LWW priors) when no session has ever existed — mergeInto:286-290 already re-queues all local-only records on first sign-in, so nothing is lost — and delete st[k] after a confirmed server delete in pushNow. _[sync engine, localStorage]_
- **Test gap: the dangerous sync branches and the RequireSurvey gate are unpinned** — src/lib/sync.test.js (100 lines) covers records/reassemble/diff and three mergeInto LWW cases only. Unpinned: the delete-mirroring branch (sync.js:276-285), the local-only re-queue branch (286-290), the pushNow dirty-queue lifecycle and its in-flight race, pulledAt semantics, and RequireSurvey's four boot states (local-hit, unconfigured, no-session, probe-error) — exactly the code that fires when sync resumes after months. Add _internals-driven mergeInto tests for empty-server-pull and re-queue, a mocked-supabase pushNow test asserting flags survive a mid-flight onLocalWrite, and a component test for RequireSurvey's error→'need' fallback and (once added) its timeout. _[tests]_

## performance — 84/100

The hygiene fundamentals are genuinely good — every listener is paired with a removal, every timer is wall-clock anchored and cleaned up, animations are opacity/transform-only under a reduced-motion kill switch, and the paused-Supabase path correctly short-circuits sync so shipping mode has zero idle network or subscriber overhead. But the deck's hot path does redundant O(entire-history) JSON work: the engagement sidecar is re-parsed and re-stringified in full on every card impression, every tap, and every summary render, its per-day records are never pruned, and the 20s shell clock re-renders the whole unmemoized tree — including two localStorage JSON.parse calls that sit directly in render bodies. Nothing is broken today at day-1 data sizes, which is exactly why these compounding costs need to be fixed before real users accumulate months of history on old iPhones.

**Strengths:**
- Event-listener and interval hygiene is flawless — every addEventListener in app code has a matching removal in the effect cleanup, and all four timers (App 20s clock, Sprint 250ms, DeepWork/Outlast 1s, guardian 15min) are gated on running state, cleared on unmount, and wall-clock anchored so background throttling can't drift them.
- Animation discipline: all six keyframes animate only opacity/transform (compositor-friendly, no paint-triggering properties), the 'motion ceiling' caps what may pulse, and a global prefers-reduced-motion block (index.css:563-570) neutralizes everything.
- Shipping-mode fast path: sync.start() returns before subscribing when Supabase is unconfigured, so storage.set() skips the readRaw prev-value parse entirely (storage.js:155-156) — the paused backend costs nothing at runtime.
- Font strategy is right for WKWebView: sans/serif ride system fonts, only IBM Plex Mono 400/500 is bundled, subsetted with unicode-range so devices fetch ~28KB of latin woff2; lucide-react is tree-shaken to zero bytes in the output.
- React 18 auto-batching plus stable block keys mean the recordSeen mount storm collapses into few render passes, and BlockRenderer's [id]-keyed effect correctly avoids re-firing impressions on mere re-renders.

**NOW:**
- **localStorage JSON.parse runs in the render body of useLayout and useRecord** — layout.js:66 (`const initial = pickInitial()`) and record.js:76 (`const initial = loadCache()`) execute on EVERY render of LiveDeck, not just mount — pickInitial does a localStorage.getItem + JSON.parse + full normalizeLayout of the cached layout payload, loadCache parses the checkpoint cache. LiveDeck re-renders every 20s (App clock) and on every engagement notify (every tap), so this synchronous parse work runs ~3+/min while the deck idles, for a value React discards after the first render. Fix is one line each: move the call into the lazy initializer — `useState(() => pickInitial().layout)` / `useState(() => loadCache().rows)` — hoisting a single `const initial` inside the initializer closure. _[deck]_

**BEFORE LAUNCH:**
- **Engagement sidecar: unbounded day growth + full-blob parse/stringify on every interaction** — engagement.js never prunes `state.days` (dayRec at engagement.js:63-66 only ever adds; no delete anywhere in the file), and every recordSeen/recordUse/completeImpact does read() → JSON.parse of the ENTIRE history then write() → JSON.stringify of it all (engagement.js:43-59, 71-98). Each tap additionally triggers summarizeDay's own read() in useEngagementSummary (engagement.js:169, 293) in both App.jsx:103 and LiveDeck.jsx:233, plus RefactorIndicator's getRefactorState — 3-4 full parses + 1 full stringify per tap. At ~1KB/day this is ~180KB of JSON re-parsed per interaction after six months, on the main thread of an old iPhone, per tap. Two fixes: (1) in closeDay(), delete days older than ~30 app-days after stashing the payload; (2) add a module-level cache to read() (set on write(), since this module is the key's only writer) so repeat reads cost nothing. _[deck, engagement telemetry]_
- **The 20s shell clock re-renders the entire app tree and rebuilds the whole deck layout** — useClock's setInterval (App.jsx:53) sets state on App itself, so every 20 seconds the full unmemoized tree re-renders: LiveDeck re-runs buildLiveLayout over every store slice + normalizeLayout + applyReadiness/applyCounsel/applyGuardian + counselFor/strongestPattern (LiveDeck.jsx:299-341), LayoutHost re-renders every widget, and the two render-body parses from the finding above fire. Fix: extract the clock into its own `<ClockDisplay/>` component (it renders one <span>, App.jsx:184) so the interval's state lives there, and wrap LiveDeck's layout pipeline in useMemo keyed on [store slices used, summary, drift, readiness, counsel] — the transforms are already pure/immutable so memoization is safe. _[deck, status strip]_
- **Every engagement event runs a full Guardian assessment synchronously behind it** — useDriftSentinel subscribes tick directly to the engagement pub/sub (guardianEngine.js:456), so every recordSeen (N of them on each return to the deck tab, since App.jsx:194 unmounts/remounts LiveDeck and BlockRenderer.jsx:54-57 fires per mount) and every recordUse tap runs runAssessment (guardianEngine.js:302-322): readToday, foldBaseline, storage.get('streak') + storage.get('wellness') parses, another summarizeDay full engagement parse, and a writeGuardian localStorage WRITE for the heartbeat — 4-5 extra parse/stringify cycles per tap, times N blocks on tab return. Debounce the subscription tick (e.g. 250ms trailing) and skip the writeGuardian heartbeat when band and day are unchanged; the 15-min interval already covers the temporal vector. _[deck, guardian]_
- **Batch the recordSeen mount storm into one storage round-trip** — Returning to the DECK surface (or switching payload tabs in LayoutHost) remounts every BlockRenderer, and each fires recordSeen individually (BlockRenderer.jsx:54-57) — with ~8 blocks that is 8 sequential full parse+stringify cycles of the engagement blob plus 8 notify fan-outs during the mount frame. Keep the per-remount impression semantics (the comment says it's intended) but coalesce the writes: have recordSeen push into a pending array flushed in one queueMicrotask — one read(), N mutations, one write(), one notify(). Combined with the read-cache fix this makes tab switching O(1) in storage work instead of O(blocks × history). _[deck]_

**FUTURE:**
- **Single 603KB JS chunk: code-split routes and lazy-load @supabase/supabase-js** — The build emits one 603KB minified / 177KB gz chunk (dist/assets/index-*.js) — no React.lazy anywhere (grep confirms), so Waitlist.jsx, Onboarding.jsx, and the full @supabase/supabase-js client (GoTrue + Realtime present in the chunk; Supabase is PAUSED in shipping mode) are parsed at every cold launch. - **Trim the service-worker precache inside the Capacitor wrap** — The SW precaches 34 entries / 815KB (vite.config.js globPatterns includes all woff2+woff), duplicating the app's local files — including cyrillic/cyrillic-ext IBM Plex subsets no English-locale user ever renders and legacy .woff beside every .woff2 — into Cache Storage on first launch inside a WKWebView that already serves from disk. Either exclude the SW from the Capacitor build (keep it for the PWA web target only) or narrow globPatterns to latin woff2 subsets; also consider dropping `backdrop-blur` from the three permanent chrome layers (App.jsx:163, BlockRenderer.jsx:114, NavBar.jsx:51) — their bg is 95% opaque so the blur is nearly invisible while still costing a compositing pass over every scroll frame on low-end devices. _[cold launch, chrome]_

## state-integrity — 82/100

The single-choke-point architecture is real and mostly honored: every model read passes through sanitize(), every sidecar read is try/catch-defensive, the pre-boot theme script is guarded, wipeAll enumerates the prefix instead of a hardcoded list, and the export deliberately strips relapse-confession drafts. But the edge paths where a book is actually lost are the weak ones: restore-from-backup silently deletes private drafts, quota-full and corrupt-slice failures are console.warn-only (a corrupted streak silently resets to day 0), wipe leaves native daily notifications firing on a "fresh start" phone, and storage.js itself has zero unit tests — the round-trip, wipe, and quota behaviors are pinned by nothing.

**Strengths:**
- One choke point that mostly holds: storage.js get() sanitizes on EVERY read (storage.js:147), object stores merge over defaults, and array-corrupted-to-primitive falls back cleanly (storage.js:131-140)
- Every sidecar reader (purchases.js:63-72, guardianEngine.js:43-51, engagement.js:43-51, analytics.js:20-30, LiveDeck.jsx:92-99, Paywall.jsx:36-48, SettingsSheet.jsx:259-266, index.html:21-31) wraps JSON.parse in try/catch with a sane default — no render-crash path from a corrupt sidecar found
- Privacy-by-construction is coherent: exportAll strips handover drafts (storage.js:188), wipeAll enumerates the whole lockedin: prefix keeping only the Apple-owned entitlement (storage.js:209-229), guardianSeed merges without clobbering engine bookkeeping (guardianSeed.js:81-96)
- StrictMode hazards were actually thought about: sync.start is idempotent (sync.js:347-350), initStreak/initSettings/initTasks are idempotent initializers, and recordInvocation has an explicit 5s double-mount dedupe guard (guardianEngine.js:76-84)
- Write-path clamping is duplicated at the action layer (addIncome rejects negatives store.jsx:283, addRun clamps store.jsx:296-306), so sanitize is a second net, not the only one

**NOW:**
- **Restoring a backup silently deletes the user's private Handover drafts** — exportAll strips drafts by design (storage.js:188 — drafts:[]), but importAll (storage.js:193-199) then writes that emptied handover slice verbatim over the live one, and importData (store.jsx:419-433) re-reads it into React.
